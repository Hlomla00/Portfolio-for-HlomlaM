## Reflection: Coding in Markdown (STAR Method)

**Situation**

As part of the PRP370S Project Presentation 3 module at CPUT, I was required to build and document a professional digital portfolio using GitHub and Markdown. I was in my final year of the Diploma in ICT: Application Development and needed to present my skills, projects, and entrepreneurial journey in a way that was accessible, professional, and readable directly on GitHub.

**Task**

My task was to write and structure the entire portfolio content in GitHub Flavoured Markdown — including my CV, project descriptions, academic history, and reflections — and deploy it to GitHub Pages. This required learning a new syntax I had not formally used before and applying it consistently across all sections.

**Action**

I treated this the same way I treat any new technical challenge: I read the documentation before writing a single line. I studied GitHub Flavoured Markdown syntax, practised formatting tables, headings, badges, and embedded links, and structured my CV as a clean, readable document. I applied the same principle I use when building apps — understand the brief fully before executing. I connected each section back to something real: my projects SafeRide and GUM, my hackathon experience, my entrepreneurial background.

**Result**

I successfully produced a complete, professional CV and portfolio in Markdown that renders cleanly on GitHub. The experience taught me a practical skill I now use daily for README files, documentation, and project communication. More importantly, it taught me that documentation is not separate from the work — it is part of it.

## Reflection: Mock Interview Experience (STAR Method)

**Situation**

As part of work readiness preparation at CPUT, I was required to participate in a mock interview simulating a real job interview for a Full-Stack Developer WIL position. I am in my final year and actively seeking a Work Integrated Learning opportunity where I can apply the full-stack skills I have developed building SafeRide, GUM, and other applications.

**Task**

I needed to prepare for and record a mock interview that demonstrated my technical knowledge, communication skills, and ability to present myself professionally to a potential employer — someone who would be evaluating not just what I know, but whether I can articulate it clearly under pressure.

**Action**

I prepared by connecting my answers to real experiences. When asked about my projects, I did not just describe features — I explained the problems they solve and why those problems matter. I talked about standing in queues at Home Affairs, about people without smartphones needing rides, about vouchers expiring because banks do not talk to each other. I rehearsed using the STAR method and focused on being specific rather than generic. I also reflected on what the Entrepreneurship Game taught me: that energy without direction is not a strategy, and that the same applies to interview preparation.

**Result**

The mock interview significantly improved my confidence. I identified areas where I needed to be more concise and where I needed to use more precise technical vocabulary. It was invaluable preparation for real WIL and graduate interviews. It also made me realise that my story — from a mobile car wash in December 2023 to building apps — is compelling if I tell it with clarity and purpose.

## Published on GitHub Pages

## Reflection: GitHub Pages (STAR Method)

**Situation**

After building my portfolio locally using React, TypeScript, Vite, and Tailwind CSS, I needed to deploy it publicly so that lecturers, moderators, and potential WIL employers could access it from any device via a live URL. I had built real projects — SafeRide, GUM, QueUp, VoucherBridge — and I needed a platform that would showcase them properly, not just store them.

**Task**

My task was to configure and deploy my React and Vite portfolio to GitHub Pages, ensuring it built correctly, loaded all assets, and was accessible and professional. This was not just a technical requirement — it was the difference between having a portfolio and having a presence.

**Action**

I configured the Vite `base` path for GitHub Pages (`/pixel-cinematic-code/`), added a `predeploy` and `deploy` script to `package.json`, and created a GitHub Actions workflow (`.github/workflows/deploy.yml`) that automatically builds and deploys the site on every push to `main`. I resolved asset path issues that caused a blank page on initial deployment and treated each deployment error the same way I treat a bug in my code — as information telling me where to look, not as a failure.

**Result**

My portfolio is now live on GitHub Pages, publicly accessible, and automatically updated every time I push new code. The experience gave me hands-on knowledge of the full deployment pipeline — from local development to production — and reinforced the principle I carry into every project: it is not finished until it is shipped. That instinct started with a mobile car wash in December 2023 and it drives everything I build today.
